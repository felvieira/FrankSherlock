# v0.8.0

### Bug fixes

- **Fix Windows startup crash** (#1): GUI apps spawning subprocesses (nvidia-smi, python, ffmpeg) caused visible console windows to flash in a loop on Windows. All 14 subprocess call sites now use `CREATE_NO_WINDOW`. Added a 5-second timeout for nvidia-smi to prevent hangs on broken/updating drivers. GPU and Python detection are now deferred to first use (lazy `OnceLock`), so the window appears instantly. Replaced a `.expect()` panic on database init failure with a graceful error exit.

- **Fix model detection with `:latest` tag** (#2): When users install the model via `ollama pull qwen2.5vl` (without an explicit tag), Ollama stores it as `qwen2.5vl:latest`. The setup screen was doing an exact string match against `qwen2.5vl:7b`, asking the user to re-download an already-installed model.

- **Fix blank screen when adjusting near-duplicates threshold** (#3): Dragging the near-duplicates percentage slider caused the screen to go blank. The duplicates data was being cleared to `null` before the new API call completed, which unmounted the entire DuplicatesView component. Previous results now stay visible while re-fetching.

- **Fix ffmpeg not detected on macOS** (#4): macOS GUI apps launched from Finder or the Dock don't inherit the user's shell `$PATH`, so Homebrew-installed ffmpeg/ffprobe at `/opt/homebrew/bin` were invisible. Added well-known fallback path probing, matching the existing approach for Python detection.

- **Fix classification failing with different model variant** (#5): When the app recommends `qwen2.5vl:7b` but the user has a different variant installed (e.g. `:3b` or `:latest`), classification would silently fail because Ollama was asked for a model tag that didn't exist. The scan now resolves the actual installed model tag, and the setup screen accepts any variant from the same model family.

### Test suite
